\section{Setup}

All experiments were conducted on a machine \textcolor{red}{Please adjust this value with the computer which has produced the results} with an 12-core AMD Ryzen 9 5900X CPU, 32-GB of RAM, and a 1-TB NVMe SSD running Ubuntu 22.04 LTS. 
The implementation and all subsequent analyses were carried out in Python 3.11.13.

\subsection{Software Environment and Dependencies}

The entire codebase is written in Python and relies on three core libraries: 
\begin{itemize}[nosep]
	\item \texttt{numpy} for numerical operations 
	\item \texttt{matplotlib} for plotting
	\item \texttt{pandas} for data handling. 
\end{itemize}

\subsection{Reproducibility Measures}

To guarantee that the experiments are fully reproducible, the following measures were taken:

\begin{itemize}[nosep]
    \item A fixed random seed (\texttt{np.random.seed(42)}) is used in the data generation script and in every solver that relies on pseudo-random numbers 
	(e.g., random 2-Opt moves, tournament selection, mutation). 
	This ensures that all stochastic decisions are deterministic across runs.
    \item The exact version of each dependency is pinned in \texttt{pyproject.toml}, and the environment is managed by \texttt{uv}.
    \item The complete source code is included in the supplementary material, 
	and the command lines for generating instances, running the benchmark, 
	and producing plots are documented in the README.
\end{itemize}

\subsection{Running the Benchmark}

The benchmark is executed via the script \texttt{src/benchmark.py}. 
It automatically discovers all problem instances in the \texttt{data/} directory and runs the selected solvers in parallel using Python's \texttt{ProcessPoolExecutor}. 
The number of parallel workers can be controlled with the \texttt{--workers} flag (defaults to the number of CPU cores). 
Already computed results are skipped to allow incremental runs.

All results are appended to \texttt{results/results.json} and also exported to \texttt{results/results.csv} for easier analysis. 
The base solver class automatically records the algorithm name, problem path, problem size, instance type, tour cost, solving time, a validity flag, and a timestamp.

\subsection{Obtaining Optimal References}

The optimal tour lengths (or best known lower bounds) are obtained using the \textsc{Concorde} TSP solver. 
Concorde is a widely used exact solver that implements cutting-plane methods and branch-and-bound. 
The binary was downloaded from the official Concorde website, placed in \texttt{bin/}, and made executable. 
The wrapper \texttt{src/algorithms/concorde\_solver.py} invokes Concorde on each instance and parses its output. 
The resulting optimal costs serve as the ground truth for computing optimality gaps.

\subsection{Post-processing and Plotting}

After all benchmark runs, aggregated statistics
(mean and standard deviation of tour cost, solving time, and optimality gap per algorithm, problem size, and instance type) 
are computed by \texttt{src/utils/compute\_statistics.py}. 
The plots shown in Section \ref{sec:results} (time scaling, solution quality, gap bars, and heatmaps) are generated by \texttt{src/utils/create\_aggregate\_plots.py}.


\subsection{Dataset Generation}

To evaluate the performance of the TSP algorithms a synthetic benchmark dataset of Euclidean TSP instances is generated. 
The generation process is controlled by a Python script that creates two types of point distributions: \textit{uniform} and \textit{clustered}. 
The script uses a fixed random seed (\texttt{42}) to ensure reproducibility.

\subsubsection{Instance sizes and repetitions}
For each problem size \(n \in \{10, 25, 50, 100, 200, 500\}\) 30 independent instances were generated. 
The same set of instances is used for all algorithms, allowing a fair comparison.

\subsubsection{Uniform instances}
Uniform instances are created by drawing \(n\) coordinates independently from a uniform distribution over the unit square \([0,1]^2\). 
The Euclidean distance between every pair of coordinates is computed to form a symmetric distance matrix.

\subsubsection{Clustered instances}
Clustered instances are designed to mimic real-world scenarios where coordinates of e.g. cities or harbors form clusters.
The number of clusters is set to \(\max(1, n // 10)\). 
Cluster centers are sampled uniformly from \([0.1, 0.9]^2\) (avoiding the extreme borders). 
Each node is assigned to a center in a round-robin fashion
and its coordinates are drawn from a normal distribution around that center with a standard deviation of \(0.05\). 
The resulting coordinates are clipped to the unit square to keep all points inside \([0,1]^2\). 
The distance matrix is then computed as for the uniform case.
