\section{Setup}
\paragraph{Benchmark Instance Generation}
To compare the different algorithms presented in the previous section, a simple benchmark is with synthetic TSP instances is created.

For each problem size \(n \in \{10, 25, 50, 100, 200, 500, 800, 1000, 1500, 2000\}\) 30 instances were generated per city distribution type.

The first set was created by sampling the cities coordinates uniformly from the unit square \([0,1]^2\).

The second set was created to simulate clustered cities, this is more realistic, because due to terrain etc. cities are not distributed uniformly on the map.
For each problem instance
\begin{enumerate}
	\item the number of cluster is set depending on the problem size to \( \max(1, n // 10) \).
	\item Cluster centers are sampled uniformly from \([0.1, 0.9]^2\) to avoid the boarders.
	\item Each subsequent node is drawn from a normal distribution around one of the cluster centers with \( \sigma = 0.05 \) until \( n \) is reached.
	      An equal amount of nodes are drawn around each cluster center.
\end{enumerate}

\paragraph{Benchmark Ground Truths}
The optimal tours for the benchmark instances were obtained using the Concorde TSP solver. After computing the optimal tour, the tour length was recorded for later comparison.

\paragraph{Baselines}
In addition to the above described algorithms and the ground truth computed by Concorde, we also provide a greedy nearest neighbor baseline for comparison.
This is the most simple way to build a tour. Starting from some city, the next city to be visited is always the one that is closest to it, this is repeated until all cities have been visited.


\paragraph{Hardware \& Environment}
All experiments were conducted on a machine with a 32-core (64-thread) Intel Xeon Gold 6326 CPU, 504 GB of RAM.



\section{Results}\label{sec:results}
In this chapter we present the results of the discussed algorithms in addition to the greedy baseline and Concorde on the benchmark instances.
Figure \ref{fig:optimality_gap_lines_uniform} shows  the optimality gap on the uniform instances along with the standard deviation across the 30 instances per size, figure \ref{fig:optimality_gap_lines_clustered}.
In general, the gap is higher on the clustered instances, than on the uniform sampled instances across all algorithms. However, this is mostly visible on the smaller sized instances with less than 500 nodes and their respective gaps converge on larger instances.
One exception to that is simulated annealing, that performs worse on the uniform instances than on the clustered ones for size 200.
Another one is tabu search on size 100 that also performs worse on the uniform instances.

\begin{figure}
	\begin{center}
		\includegraphics[width=0.95\linewidth]{figures/optimality_gap_lines_uniform.png}
	\end{center}
	\caption{Optimality gap of uniformed algorithms for different sizes and distributions}\label{fig:optimality_gap_lines_uniform}
\end{figure}
Iterated local search consistently performs the best across all sizes and distributions, followed by tabu search and 2-opt, where tabu search dominates for instances smaller than 500 but drops in performance on instances bigger than that.
They are followed with a big margin by simulated annealing and the genetic algorithm, where simulated annealing wins until size 200 but looses starting from size 500.
Consistently the worst one is the greedy algorithm, although performance is pretty much the same as simulated annealing starting from size 500.

Interestingly, for most of the algorithms the gap is improving again after size 500, it plateaus there and then improves again.

When looking at the time scaling in figures \ref{fig:time_scaling_uniformed} , \ref{fig:time_scaling_clustered},  it looks very similar between uniform and clustered instances again. There is a small difference between them for Concorde at 200 nodes and for tabu search at size 2000.

Greedy is the fastest by a big margin, followed by 2-opt for up to size 1000 before simulated annealing runs faster than it.
After 2-opt follows the exact Concorde solver for most of the sizes, it is not directly comparable though because it was written in C instead of python.
Afterwards iterated local search, tabu search, genetic algorithm and simulated annealing follow relatively close to each other, but they switch order after size 100.

Overall, although it also requires the most time for small instances, simulated annealing exhibits the best scaling behaviour, even running faster than Concorde and 2-opt for large instances.
